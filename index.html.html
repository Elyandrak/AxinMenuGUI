AxinMenuGUI

⚔ AxinMenuGUI

Crea menús GUI completamente configurables para tu servidor VintageStory — sin escribir código.

AxinMenuGUI es un mod de menús interactivos para VintageStory que permite a los administradores construir menús complejos y multipágina mediante ficheros JSON. Sin programar, sin reiniciar el servidor.

Inspirado en GUIPlus (Minecraft/Spigot), diseñado desde cero para la API de VintageStory y la metodología AXIN.

🖼
100% JSON
Crea y edita menús con ficheros JSON. Recarga en caliente sin reiniciar.
🖱
16 Click Events
Mensajes, comandos, dar/quitar ítems, teletransporte, navegar entre escenas y más.
✅
Condiciones
Muestra u oculta botones según permisos, ítems en inventario, cooldowns y variables.
📄
Multi-escena
Menús multipágina con navegación independiente por escena.
💾
Player Data
Variables por jugador con operaciones matemáticas. Puntos, rangos, contadores.
⚡
Múltiples triggers
Abre menús por comando, ítem, hotkey, al conectarse, o al entrar en una zona.
💬
Chat Fetcher
Solicita input al jugador via chat y úsalo en acciones dinámicas.
🔗
Tokens dinámicos
Usa {player}, {var:campo} y más en cualquier texto del menú.

📋 Requisitos

RequisitoValor
VintageStory1.21.x+
Tipo de modUniversal (cliente y servidor)
DependenciasNinguna (mod standalone)
Integración opcionalAxinClaimsRules (trigger de zona, Fase 3)
⚠
Distribución al cliente — obligatoria

AxinMenuGUI ejecuta código en el lado del cliente y renderiza GUI. Debe estar instalado en el cliente además del servidor. En VintageStory esto significa publicarlo en ModDB o distribuirlo manualmente — el servidor no lo envía automáticamente.

🚀 Inicio rápido

Instala AxinMenuGUI y crea tu primer menú en minutos.

📥 Instalación

Descarga el .zip del mod e instálalo como cualquier mod de VintageStory:

  1. Copia el fichero AxinMenuGUI_vX.Y.Z.zip en la carpeta Mods/ del servidor.
  2. Reinicia el servidor. El mod generará sus ficheros de configuración.
  3. Comprueba en los logs: [AxinMenuGUI] Mod cargado. X menús encontrados.
💡
El mod es Universal: también debe estar presente en el cliente. VintageStory lo descargará automáticamente si está publicado en ModDB.

📂 Ficheros generados

# Estructura de ficheros tras el primer inicio
ModConfig/AxinMenuGUI/
├── menus/                 # Un fichero .json por menú
│   └── example.json       # Menú de ejemplo generado automáticamente
└── playerdata/            # Variables por jugador (se crea al necesitarse)
    └── {uid-sanitizado}.json
ℹ
No hay fichero Config.json global. Toda la configuración vive en los propios JSON de cada menú dentro de menus/.

🎯 Tu primer menú

Crea un fichero ModConfig/AxinMenuGUI/menus/hola.json:

hola.json{
  "id": "hola",
  "title": "Mi primer menú",
  "rows": 3,
  "commandAlias": "hola",
  "scenes": {
    "0": {
      "items": {
        "1": {
          "slot": 13,
          "item": "game:gem-diamond",
          "name": "Hola, {player}!",
          "lore": ["Haz clic en mí"],
          "clickEvents": {
            "saludo": {
              "type": "message",
              "message": "¡Bienvenido, {player}!"
            }
          }
        }
      }
    }
  }
}

Recarga sin reiniciar:

/amenu reload

Abre el menú:

/hola   # alias directo
/amenu open hola   # comando estándar

📋 Comandos y permisos

Todos los comandos disponibles y sus nodos de permiso.

📋 Comandos

El comando base es /amenu. Cada menú puede definir un alias propio con el campo "commandAlias" en su JSON.

ComandoDescripciónPermiso
/amenu open <id>Abre un menú para el propio jugador—
/amenu open <id> <jugador>Abre un menú para otro jugadorcontrolserver
/amenu reloadRecarga todos los JSONs sin reiniciarcontrolserver
/amenu listLista todos los menús cargadoscontrolserver
/amenu info <id>Muestra detalles de un menúcontrolserver
/<alias>Atajo directo (definido por menú con commandAlias)Permiso del menú
/<alias> <jugador>Abre el menú para otro jugador (según commandAliasTarget)controlserver

🔑 Permisos

ℹ
VintageStory no tiene un gestor de permisos externo como LuckPerms. Los privilegios se asignan por rol en el serverconfig.json del servidor. El campo "permission" de un menú comprueba si el jugador tiene ese privilegio via la API nativa player.HasPrivilege().
PermisoDescripción
controlserverPrivilegio nativo VS requerido para reload, list, info, y abrir menús para otros jugadores
(por menú)Permiso definido en el campo "permission" del JSON del menú. Cualquier string de privilegio VS válido — el admin lo asigna al rol en serverconfig.

🔐 Permisos por menú

Cada menú puede requerir un privilegio específico. El jugador que no lo tenga no podrá abrir el menú:

vipshop.json{
  "id": "vipshop",
  "title": "Tienda VIP",
  "permission": "axinmenu.vip",
  ...
}

Para que funcione, el rol correspondiente en tu serverconfig.json debe incluir "axinmenu.vip" en su lista de Privileges.

⚙ Configuración

AxinMenuGUI no tiene fichero de configuración global. Todo se configura por menú.

ℹ
Sin Config.json global

A diferencia de otros sistemas de menús, AxinMenuGUI no usa un fichero de configuración central. Cada menú es autónomo y se configura completamente dentro de su propio .json en ModConfig/AxinMenuGUI/menus/.

📂 Dónde va cada cosa

ElementoUbicación
Definición de menúsModConfig/AxinMenuGUI/menus/*.json
Variables por jugadorModConfig/AxinMenuGUI/playerdata/{uid}.json (auto-gestionado)
Alias del comandoCampo "commandAlias" dentro del JSON de cada menú
Permiso de accesoCampo "permission" dentro del JSON de cada menú
Triggers de aperturaCampo "openTriggers" dentro del JSON de cada menú

🔄 Recarga en caliente

Puedes editar cualquier fichero JSON y recargar sin reiniciar el servidor:

/amenu reload   // Recarga todos los menús desde disco

Los cambios son inmediatos. No se necesita reiniciar ni desconectar a los jugadores.

📝 Formato de nombres de fichero

Los ficheros de menú pueden llamarse con cualquier nombre. El ID del menú se define dentro del JSON con el campo "id", que es el que se usa en los comandos:

// Fichero: ModConfig/AxinMenuGUI/menus/mi-fichero.json
{
  "id": "mainmenu",   // Este es el ID que usa /amenu open mainmenu
  ...
}

🖼 Estructura del GUI

Define menús personalizados mediante ficheros JSON.

Cada menú es un fichero .json en ModConfig/AxinMenuGUI/menus/. El nombre del fichero es libre; el ID del menú se define dentro.

📐 Estructura completa

Esta es la estructura real generada por el mod. Los campos vacíos ("", [], {}) son opcionales — solo incluye los que necesites.

ejemplo.json{
  "id": "ejemplo",              // ID único del menú (requerido)
  "title": "Menú de Ejemplo",   // Título mostrado al jugador
  "rows": 3,                  // Filas del grid (define el tamaño de la ventana)
  "commandAlias": "ejemplo",  // /ejemplo → abre este menú
  "commandAliasTarget": "OPTIONAL", // OPTIONAL | REQUIRED | DISABLED
  "permission": "",           // Permiso requerido (vacío = todos)
  "openTriggers": [],         // Triggers adicionales (ver Triggers)
  "scenes": {
    "0": {                     // Primera escena/página (índice 0)
      "delay": 0,             // Reservado (presente en modelo, no funcional aún)
      "items": {
        "bienvenida": {       // ID del ítem (único dentro de la escena)
          "slot": 13,          // Posición en el grid (0-indexed)
          "item": "game:gem-diamond",
          "amount": 1,
          "name": "Hola, {player}!",
          "lore": ["Línea 1", "Línea 2"],
          "clickEvents": {
            "saludo": {
              "type": "message",
              "clickType": "ANY",    // LEFT | RIGHT | ANY
              "delay": 0,            // ms antes de ejecutar
              "message": "¡Bienvenido, {player}!",
              "commands": [],        // para consoleCommand / playerCommand
              "item": "",            // para giveItem / takeItem
              "amount": 1,           // para giveItem / takeItem
              "location": "",        // para teleport: "x,y,z"
              "guiId": ""            // para openGui: id del menú destino
            }
          },
          "conditions": {},       // vacío = sin condiciones
          "conditionFailMessage": ""
        }
      }
    }
  }
}

🏷 Propiedades del menú

CampoTipoDescripción
idstringIdentificador único del menú. Se usa en comandos y en openGui. (requerido)
titlestringTítulo visible en la ventana del GUI
rowsintNúmero de filas del grid (columnas fijas = 9)
commandAliasstringAlias de comando directo: /alias abre este menú
commandAliasTargetstringOPTIONAL = acepta jugador opcional · REQUIRED = requiere jugador · DISABLED = sin argumento
permissionstringPrivilegio VS requerido para abrir el menú. Vacío = acceso libre.
openTriggersarrayTriggers adicionales de apertura (ver Triggers). Vacío = solo por comando.
scenesmapEscenas del menú, indexadas por número ("0", "1"…). Requerido.

🏷 Propiedades del ítem

CampoTipoDescripción
slotintPosición en el grid (0 = arriba-izquierda)
itemstringCódigo de ítem VS: game:gem-diamond, game:stone…
amountintCantidad (default: 1)
namestringNombre mostrado. Admite tokens {player} etc.
lorestring[]Líneas de descripción bajo el nombre
clickEventsmapAcciones al hacer clic (ver Click Events)
conditionsmapCondiciones de acceso (ver Condiciones)
hideOnFailbooltrue = ítem invisible si condición falla · false (default) = visible pero clic bloqueado
conditionFailMessagestringMensaje al jugador si una condición falla (solo si hideOnFail: false)

📄 Escenas

Crea menús multipágina con layouts independientes por escena.

Las escenas permiten tener múltiples páginas dentro de un mismo menú. Cada escena tiene su propio layout de ítems y puede navegarse con click events. Los índices empiezan en 0.

📐 Estructura de escenas

shop.json (2 páginas){
  "id": "shop",
  "title": "Tienda — Página {scene}",
  "rows": 3,
  "scenes": {

    "0": {                          // Página 1 (slots 0-26 en grid 3×9)
      "items": {
        "next": {
          "slot": 26,              // última celda de la fila 3
          "item": "game:arrow-flint",
          "name": "Siguiente >",
          "clickEvents": {
            "nav": { "type": "nextScene" }
          }
        }
      }
    },

    "1": {                          // Página 2
      "items": {
        "prev": {
          "slot": 18,              // primera celda de la fila 3
          "item": "game:arrow-flint",
          "name": "< Anterior",
          "clickEvents": {
            "nav": { "type": "previousScene" }
          }
        }
      }
    }
  }
}

🏷 Propiedades de escena

CampoTipoDescripción
delayint (ms)Reservado — presente en el modelo. Funcionalidad planificada para fases futuras.
itemsmapÍtems de esta escena (requerido)
ℹ
Las escenas son páginas independientes del menú. La navegación entre ellas se hace con los click events nextScene, previousScene, openGui y back.

🧭 Click Events de navegación

typeDescripción
nextSceneAvanza a la siguiente escena
previousSceneRetrocede a la escena anterior
openGuiAbre otro menú diferente
backReabre el último menú visto
closeGuiCierra el menú

🎨 CREAR GUIs › Click Events

🖱 Click Events

Acciones que se ejecutan cuando un jugador hace clic en un ítem del menú.

Un ítem puede tener múltiples click events que se ejecutan en secuencia. Se definen en el map clickEvents del ítem, con una clave única por evento.

ℹ
El mod serializa todos los campos del modelo en cada click event, aunque estén vacíos. En los ejemplos de esta página solo se muestran los campos relevantes para cada tipo — omitir el resto es perfectamente válido.

📋 Referencia rápida

messageEnvía mensaje de chat al jugadorv1
consoleCommandEjecuta comando como servidor (sin restricciones de permiso)v1
playerCommandEjecuta comando como el jugador (con sus permisos)v1
closeGuiCierra el menúv1
openGuiAbre otro menúv1
nextSceneVa a la siguiente escenav1
previousSceneVa a la escena anteriorv1
backReabre el menú anteriorv1
buyItemIntercambio de compra: el jugador paga ítems y recibe ítemsv2

⚙ Propiedades del modelo

Estos son todos los campos que el mod serializa en un click event. En la práctica solo necesitas type y los campos relevantes a ese tipo — el resto puede omitirse o dejarse vacío.

CampoTipoDefaultUsado por

🖱 Tipos de eventos detallados

message — Enviar mensaje

{
  "type": "message",
  "message": "Hola, {player}! Bienvenido."
}

consoleCommand — Comando de consola

Ejecuta uno o varios comandos como si los escribiera la consola del servidor (sin restricciones de permiso del jugador). El token {player} se expande al nombre del jugador.

{
  "type": "consoleCommand",
  "commands": [
    "/tp {player} 512000 150 512000",
    "/player {player} gamemode creative"
  ]
}

buyItem — Comprar (intercambio)

El jugador paga los ítems de cost y recibe los de give. El mod verifica que tenga suficiente stock antes de ejecutar — si no, muestra el conditionFailMessage del ítem sin hacer nada. Con "consume": false se verifica el stock pero no se consumen los ítems.

{
  "type": "buyItem",
  "consume": true,
  "cost": [
    { "item": "game:ingot-iron", "amount": 5 },
    { "item": "game:stick",      "amount": 2 }
  ],
  "give": [
    { "item": "game:sword-iron", "amount": 1 }
  ]
}

sellItem — Vender (intercambio inverso)

Semánticamente el inverso de buyItem: el jugador entrega los ítems de cost y recibe los de give. La lógica interna es idéntica — la diferencia es conceptual para que el JSON sea más legible.

{
  "type": "sellItem",
  "consume": true,
  "conditionFailMessage": "No tienes suficiente piedra.",
  "cost": [
    { "item": "game:stone", "amount": 10 }
  ],
  "give": [
    { "item": "game:ingot-iron", "amount": 1 }
  ]
}
ℹ
ItemStack — formato de ítem en cost/give

Cada entrada es un objeto con "item" (código VS) y "amount" (cantidad). Puedes incluir tantos como necesites en cada lista.

}

teleport — Teletransportar

Las coordenadas en VintageStory son absolutas. El centro del mapa por defecto está cerca de 512000, Y, 512000.

{
  "type": "teleport",
  "location": "512000,150,512000"   // x,y,z absolutos en el mapa VS
}
⚠
A diferencia de Minecraft, las coordenadas en VS no son relativas al origen del mundo. El "spawn" de tu servidor tendrá coordenadas específicas que puedes ver con /debug pos en el chat del juego.

savePlayerData — Guardar variable

{
  "type": "savePlayerData",
  "saveFormat": "puntos:{var:puntos}+10"
}

// Operadores: + - * /
// "puntos:{var:puntos}+10"  → suma 10 a puntos actuales
// "rango:vip"               → asigna string estático

🔗 Combinar múltiples eventos

Puedes combinar varios click events en un mismo clic. Se ejecutan en secuencia. Ejemplo: comprar un ítem, mostrar mensaje y cerrar el menú.

"clickEvents": {
  "compra": {
    "type": "buyItem",
    "consume": true,
    "cost": [{ "item": "game:quartz", "amount": 5 }],
    "give": [{ "item": "game:sword-copper", "amount": 1 }]
  },
  "mensaje": {
    "type": "message",
    "message": "¡Compra realizada, {player}!"
  },
  "cerrar": {
    "type": "closeGui"
  }
}
💡
Los click events se ejecutan en el orden en que aparecen en el JSON. Usa claves descriptivas para que el fichero sea legible.

✅ Condiciones

Controla la visibilidad de ítems y la ejecución de click events.

Las condiciones determinan si un ítem se muestra al jugador y si sus click events pueden ejecutarse. El comportamiento por defecto cuando falla es mostrar el ítem pero bloquear el clic con conditionFailMessage. Con "hideOnFail": true el ítem directamente no aparece.

✅ Todos los tipos de condición

typeDescripciónFase
hasPrivilegeEl jugador tiene un privilegio VS específicov1
hasItemEl jugador tiene ítem(s) en el inventariov1
hasRoleEl jugador tiene exactamente el rol indicado (por Code del serverconfig)v2
hasPrivilegeLevelEl jugador tiene nivel de privilegio igual o superior al mínimo indicadov2
playerDataCompareCompara una variable del jugador con un valorv2
cooldownCooldown por tiempo antes de poder usar de nuevov2
isGameModeEl jugador está en un modo de juego concretov2

🔍 Detalle de condiciones

hasPrivilege — Tiene privilegio VS

Comprueba si el jugador tiene un privilegio nativo de VintageStory (como build, gamemode, etc.) o uno personalizado.

"conditions": {
  "es-vip": {
    "type": "hasPrivilege",
    "privilege": "axinmenu.vip"
  }
}

hasItem — Tiene ítem

"conditions": {
  "tiene-coste": {
    "type": "hasItem",
    "item": "game:gem-diamond",
    "amount": 5
  }
}

hasRole — Tiene rol exacto

Comprueba si el jugador tiene exactamente el rol indicado. Los códigos de rol se definen en el serverconfig.json del servidor.

"conditions": {
  "es-admin": {
    "type": "hasRole",
    "roleCode": "admin"     // Códigos estándar VS: suplayer, crplayer, sumod, crmod, admin…
  }
}

hasPrivilegeLevel — Nivel mínimo de privilegio

Más flexible que hasRole: pasa si el nivel del rol del jugador es ≥ al mínimo. Útil para agrupar todos los moderadores y admins.

"conditions": {
  "es-mod-o-superior": {
    "type": "hasPrivilegeLevel",
    "minLevel": 200    // 200 = sumod · 500 = crmod · 99999 = admin
  }
}
ℹ
Los niveles de privilegio de tu servidor están definidos en serverconfig.json en el campo "PrivilegeLevel" de cada rol.

playerDataCompare — Comparar variable

"conditions": {
  "tiene-puntos": {
    "type": "playerDataCompare",
    "field": "puntos",
    "operator": ">=",       // =  !=  >  <  >=  <=
    "value": 50
  }
}

cooldown — Tiempo de espera

"conditions": {
  "espera-diaria": {
    "type": "cooldown",
    "id": "recompensa-diaria",
    "durationMs": 86400000   // 24 horas en ms
  }
}

👁 hideOnFail — Control de visibilidad

Por defecto, cuando una condición falla el ítem se muestra pero el clic queda bloqueado. Con "hideOnFail": true el ítem directamente no aparece en el menú para ese jugador:

{
  "slot": 13,
  "item": "game:gem-diamond",
  "name": "Botón solo para admins",
  "hideOnFail": true,           // invisible si la condición falla
  "conditions": {
    "solo-admin": { "type": "hasRole", "roleCode": "admin" }
  }
}
hideOnFailCondición fallaconditionFailMessage
false (default)Ítem visible, clic bloqueadoSe muestra al jugador
trueÍtem no apareceNo se muestra (ítem invisible)

🔄 Condiciones invertidas

Añade "inverted": true para negar cualquier condición:

"conditions": {
  "no-baneado": {
    "type": "hasPrivilege",
    "privilege": "axinmenu.banned",
    "inverted": true
  }
}

🔗 Múltiples condiciones (AND)

Si defines varias condiciones, todas deben cumplirse (lógica AND):

"conditions": {
  "tiene-rango": { "type": "hasPrivilege", "privilege": "axinmenu.vip" },
  "tiene-coste": { "type": "hasItem", "item": "game:gem-diamond", "amount": 5 },
  "no-cooldown": { "type": "cooldown", "id": "compra-vip", "durationMs": 3600000 }
},
"conditionFailMessage": "Necesitas ser VIP, tener 5 diamantes y esperar 1h"

💬 Chat Fetcher

Solicita texto al jugador via chat y úsalo en acciones dinámicas.

ℹ
El Chat Fetcher es una función de fase 3. Disponible en versiones avanzadas del mod. v3

⚙ Cómo funciona

  1. El jugador hace clic en un ítem con "type": "chatFetcher".
  2. El menú se cierra y aparece un mensaje de prompt en el chat.
  3. El jugador escribe su respuesta en el chat.
  4. Se valida el input contra las condiciones definidas.
  5. Si es válido, se ejecutan los click events anidados con el token {input}.
  6. Si no es válido, se muestra el conditionFailMessage.

Para cancelar: el jugador escribe cancelar en el chat (palabra fija — configurable en una versión futura).

📐 Estructura básica

{
  "type": "chatFetcher",
  "message": "Escribe algo en el chat:",
  "clickEvents": {
    "respuesta": {
      "type": "message",
      "message": "Escribiste: {input}"
    }
  }
}

✅ Con validación de input

{
  "type": "chatFetcher",
  "message": "¿Cuántos puntos quieres gastar?",
  "conditionFailMessage": "Introduce un número válido",
  "conditions": {
    "es-numero": { "type": "isInteger", "value": "{input}" },
    "tiene-puntos": {
      "type": "playerDataCompare",
      "field": "puntos", "operator": ">=", "value": "{input}"
    }
  },
  "clickEvents": {
    "gastar": {
      "type": "savePlayerData",
      "saveFormat": "puntos:{var:puntos}-{input}"
    },
    "confirmacion": {
      "type": "message",
      "message": "Gastaste {input} puntos. Te quedan {var:puntos}."
    }
  }
}

💬 INTERACTIVO › Player Data

💾 Player Data

Guarda, lee y manipula variables por jugador.

Player Data es un sistema de almacenamiento de variables por jugador. Permite implementar puntos, rangos, contadores de kills, progreso de misiones, o cualquier dato que necesites.

⚙ Cómo funciona

Los datos se guardan en ModConfig/AxinMenuGUI/playerdata/{uid}.json, un fichero por jugador:

{uid}.json{
  "uid": "abc123-...",
  "playerName": "Hector",
  "fields": {
    "puntos": 150,
    "rango": "gold",
    "kills": 42
  }
}

💾 Guardar datos

Usa savePlayerData con el campo saveFormat:

// Valor estático
"saveFormat": "rango:vip"

// Suma
"saveFormat": "puntos:{var:puntos}+10"

// Resta
"saveFormat": "puntos:{var:puntos}-{input}"

// Múltiples campos a la vez (separados por coma)
"saveFormat": "kills:{var:kills}+1,lastKill:now"

📖 Leer datos en textos

Usa el token {var:campo} en cualquier texto del menú:

"name": "Tus puntos: {var:puntos}",
"lore": [
  "Rango: {var:rango}",
  "Kills: {var:kills}"
]

🔢 Operadores matemáticos

OperadorEjemploResultado
+{var:puntos}+10Suma 10
-{var:puntos}-5Resta 5
*{var:puntos}*2Multiplica por 2
/{var:puntos}/2Divide entre 2

💬 INTERACTIVO › Tokens / Placeholders

🔗 Tokens

Valores dinámicos disponibles en textos, mensajes y comandos.

Los tokens se resuelven en el servidor en el momento del clic. Puedes usarlos en name, lore, message, y en el contenido de commands.

⚠
AxinMenuGUI es un mod de VintageStory, no de Minecraft

VS no tiene dimensiones, mundos paralelos ni PlaceholderAPI. Los tokens disponibles son los que el mod implementa explícitamente. No uses tokens de GUIPlus/Minecraft que no estén en esta lista.

TokenValorEstado
{player}Nombre del jugadorImplementado
{uid}UID único del jugador (identificador interno de VS)v2 — pendiente
{var:campo}Variable del jugador (Player Data)v2 — pendiente
{pos}Posición actual del jugador en coordenadas absolutas VS (x,y,z)v2 — pendiente
{gamemode}Modo de juego del jugador: Survival, Creative, Spectator, Guestv2 — pendiente
{scene}Índice de la escena actual (para usar en el título del menú)v2 — pendiente
{input}Texto introducido por el jugador en el ChatFetcherv3 — pendiente
{claim}Nombre del claim en el que está el jugador (requiere AxinClaimsRules)v3 — pendiente
ℹ
Nota sobre {pos} en VS

En VintageStory las coordenadas son absolutas y el origen del mapa suele estar cerca de 512000, Y, 512000. No son coordenadas relativas como en Minecraft. Úsalas con cuidado en textos orientados al jugador.

ℹ
¿Por qué no hay {world}?

VintageStory no tiene dimensiones ni mundos paralelos como Minecraft (Overworld, Nether, End). Un servidor VS es un único mundo. El token {world} no tiene equivalente en VS y ha sido eliminado de esta documentación.

Ejemplo — tokens implementados

"name": "Perfil de {player}",
"lore": [
  "Puntos: {var:puntos}",
  "Rango: {var:rango}"
],
"message": "Bienvenido, {player}! Tienes {var:puntos} puntos."

Ejemplo — token en comando

"type": "consoleCommand",
"commands": ["/player {player} gamemode creative"]

🔌 TRIGGERS

⚡ Modos de apertura

Configura cómo y cuándo se abre cada menú.

Un mismo menú puede tener múltiples triggers activos simultáneamente. El admin decide qué menú se abre en cada caso.

⚡ Tipos de trigger

TipoDescripciónFase
commandComando /alias configurado por menú. Verificado en RUN.v1
itemClic en un ítem con código específicov2
onJoinAl conectarse el jugador al servidorv3
hotkeyTecla configurable (ClientSide)v3
claimZoneAl entrar en un claim con flag específicov3

📐 Configuración

mainmenu.json{
  "id": "mainmenu",
  "commandAlias": "menu",        // Trigger tipo command (atajo rápido)
  "openTriggers": [
    {
      "type": "item",
      "itemCode": "game:feather"    // Clic derecho con una pluma
    },
    {
      "type": "onJoin"              // Al conectarse al servidor
    }
  ],
  ...
}

🗺 Trigger por zona (integración AxinClaimsRules)

ℹ
El trigger claimZone es opcional e integra con AxinClaimsRules. Si no tienes ese mod, simplemente no lo uses.
{
  "type": "claimZone",
  "claimFlag": "axinmenu.mainmenu"   // Flag definido en el claim
}
